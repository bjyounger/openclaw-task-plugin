# 开发规矩

## 核心流程

```
需求分析 → 方案制定 → CC专家组评审 → 执行开发 → 测试验证
                           ↓
                    评审不通过 → 按意见修改方案 → 重新评审
                    评审通过 → 分工给 CC 执行
```

## 关键规矩

### 1. 评审角色

**❌ 错误理解**：方案由主人评审

**✅ 正确规矩**：方案由 **CC 专家组** 评审

### 2. 评审流程

| 步骤 | 执行者 | 说明 |
|------|--------|------|
| 需求分析 | 主会话 | 梳理任务、竞品分析 |
| 方案制定 | CC 专家组 | 详细设计方案 |
| 方案评审 | CC 专家组 | 审查方案质量 |
| 评审通过 | CC 专家组 | 决定是否执行 |
| 方案修改 | CC 专家组 | 按评审意见修改 |
| 开发执行 | CC 开发专家 | 实现代码 |

### 3. 评审标准

- ✅ 架构合理
- ✅ 接口清晰
- ✅ 可实施性强
- ✅ 性能指标明确
- ✅ 测试策略完整
- ✅ 与现有模块兼容

## 专家组分工

| 专家类型 | 负责内容 |
|----------|----------|
| 架构专家 | 总体架构、模块划分、接口设计 |
| 执行专家 | 执行逻辑、算法实现 |
| 集成专家 | 模块集成、接口对接 |
| 测试专家 | 测试用例、性能基准 |
| 评审专家 | 方案审查、质量把关 |
